# 心测后续开发计划

这份文档只回答后续怎么做。当前状态请看 `STATUS.md`。

## 1. 目标

在尽量少维护文档的前提下，持续缩小“最终蓝图”与“当前实现”的差距。

## 2. 蓝图与现实

### 最终蓝图

参考：

- `xince-design-doc.docx`
- `xince-technical-design-final_121644ff.md`

目标形态：

- 内容驱动的心理测试产品
- 具备测试、报告、匹配、成长反馈、内容发现、后台管理等完整闭环

### 当前现实

- 用户端主链路已成型
- 后端主要业务域已成型
- 后台基础工作台已成型
- 内容发现、通知、星座等模块仍是明显缺口

## 3. 开发主顺序

### 阶段 1：状态持续收敛

目标：

- 以后只维护 `PROJECT.md`、`STATUS.md`、`PLAN.md`
- 新增能力时同步更新 `STATUS.md`
- 阶段结束标准：不再出现多个文档同时描述项目状态

### 阶段 2：补齐半完成产品域

优先级：

1. 发现页内容化
2. 通知消息中心接线
3. 星座内容正式后端化

结束标准：

- 这 3 个模块至少有明确的正式接口或正式下线决策

### 阶段 3：后台内容系统收敛

目标：

- 后台测试管理从只读列表升级为版本、问卷、题目、人格可编辑并可发布
- 导入预览从 JSON 提升为结构化视图
- 明确发布态如何影响用户端读取

### 阶段 4：体验增强

目标：

- 匹配体验补强
- 分享与成长反馈增强
- 设置与通知体验统一

### 阶段 5：继续向蓝图逼近

目标：

- 根据产品蓝图继续补齐差异化能力
- 优先做可被用户端、后台、后端共同消费的能力

## 4. 当前优先级

### 用户端

1. 发现页内容化
2. 通知接线
3. 星座内容接线或下线
4. 匹配体验补强
5. 分享与成长反馈收口

### 后端

1. 发现页内容接口
2. 通知消息接口
3. 星座内容接口
4. AI 与认证降级行为继续收口

### 管理后台

1. 导入预览结构化
2. 内容生命周期可视化
3. AI 工作台与系统配置增强

## 5. 执行规则

后续每次开发都遵守：

1. 改代码后同步更新 `STATUS.md`
2. 如果改动影响优先级或阶段目标，再更新 `PLAN.md`
3. `PROJECT.md` 只在入口、启动方式、文档职责变化时更新
4. 不再新增新的状态类文档，除非必须产出外部交付物
