## 1. 阶段性结论

当前项目已经从“原型资料 + 技术方案”推进到“具备生产架构雏形的 MVP 产品骨架”阶段。

从架构视角看，已经明确完成的里程碑有 6 个：

1. 内容底座完成：YAML 配置、数据库模型、版本化内容、草稿与发布已经形成闭环。
2. 用户主链路完成：测试列表 -> 测试详情 -> 答题 -> 提交 -> 报告 -> 个人中心回看已经真实可跑。
3. 报告系统完成首版：结构化报告、AI 状态流、fallback、Prompt 模板、Gateway、任务管理已经落地。
4. 超级管理员运营底座完成：导入、草稿编辑、发布控制、AI Prompt 管理和任务排障已有 API 能力。
5. 身份体系完成首版：H5 访客 token 会话、手机号演示态绑定、小程序登录参数位和提交鉴权边界已经建立。
6. 成长体系完成首版：勋章、日历、灵魂碎片、每日一问、连续奖励、分享海报已经形成可见能力。

当前项目还不是“上线收口完毕”的正式版，但已经明显越过 demo 阶段，进入“核心产品能力大体成型，剩余工作以收口、增强和上线准备为主”的阶段。

当前整体判断：

- P0 可上线 MVP 完成度：约 `99%`
- P1 增强体验完成度：约 `55%`
- P2 差异化能力完成度：约 `15%`
- 当前最接近的目标：完成 P0 收尾，让项目进入可以组织真实内测和上线准备的状态
- 预计距离 P0 收尾：约 `1-2` 轮
- 当前最主要的缺口：评分参数边界的最后收口、小程序真机联调
- 当前范围约束：短信功能当前不纳入 P0 必做，压测 / 告警 / 回滚预案当前不纳入本轮主线，匹配 / 时光胶囊 / 更重的社交玩法仍放在后续阶段

## 5. 当前主要风险与假设

### 风险

- 产品文档中的首批 8 套测试，与技术文档 / demo 当前基线不完全一致。
- H5 已切到正式 token 会话，小程序账号归并与显式升级入口也已补齐，但真实换码仍未最终收口。
- 小程序后端账号归并逻辑和前端显式升级入口已补齐，但还缺真实 `js_code` 的真机联调确认。
- 手机号验证码链路当前按演示态保留，不作为当前阶段的上线前置条件。
- onboarding 首版已经完成，但还没有更细的兴趣偏好、推荐标签和引导分流逻辑。
- 提交接口当前已经要求显式 `user_id` 必须带登录态且与当前 token 一致，避免伪造他人身份写入记录。
- 评分和报告目前能跑，但还没有覆盖更完整的参数边界和更细的异常输入约束。
- 当前已补提交完整性校验、题型级答案形态校验、数值范围校验，并支持 `rank / plot2d / colorpick` 复杂协议，但还缺更细的参数约束。
- 当前又补了配置化数值步进校验、`plot2d` 有限值 / 边界校验和提交请求基础字段约束，但还缺更细的参数约束。
- 百炼和 Ark 都已完成真实 key 冒烟，但 AI 成功率 / 时延 / 降级策略还需要继续观察。
- AI 已经补到 metrics 级别，但还没有更长期的趋势留存、告警阈值和自动化巡检。
- 成长体系当前已有勋章、日历、碎片、每日一问、连续打卡、分享卡片首版、连续奖励勋章、海报模板、独立海报页、图片预览生成和图片保存首版，但还缺多渠道分享、提醒补签和更强的复访驱动力。

### 当前假设

- 继续以 `index.html + 技术文档` 作为产品功能基线。
- 当前阶段继续优先后端和主链路闭环，不优先投入独立后台前端。
- AI 先按“可替换供应商的生产底座”推进，再做真实联调和细化运营能力。
- H5 先走“访客登录 -> token 会话 -> 后续升级绑定手机号/微信”的路线；小程序接微信换码登录。
- 手机号验证码链路保持演示态，后续若恢复需求再按阿里云 SMS 或其他供应商接入。

- 核心业务闭环已经成型，用户从进入测试到拿到报告，再到回看沉淀和成长反馈，已经能跑完整链路。
- 内容系统没有走硬编码页面路线，而是走了“YAML 配置源 + DB 运行态 + 草稿 / 发布”的可持续运营架构。
- 报告和 AI 没有直接绑死某一家模型供应商，而是已经抽象出 Prompt 模板与 Gateway，这对后续替换供应商和控制成本很重要。
- 身份体系已经从临时 `user_id` 过渡到正式 token 会话，并补上了显式 `user_id` 提交鉴权边界。

### 还没有做完的事情

- P0 还缺最后一层“上线前收口能力”：评分边界、小程序真实换码、发布前自检。
- P0 的 onboarding 首版已经补齐，主链路剩余重点进一步集中到真机联调和发布前自检。
- P1 还缺“留存增强层”：勋章进阶、日历补签、每日一问提醒、碎片叙事、多渠道分享。
- P2 基本还没真正开始：匹配、时光胶囊、AI 工作台等差异化能力仍属于后续阶段。

### 当前最重要的判断

- 现在不应该再把主要精力分散到外围功能。
- 最值得继续投入的是“让现有闭环更像正式产品”，而不是继续横向铺功能清单。
- 项目接下来最关键的成功标准，不再是“多做几个页面 / 玩法”，而是“把现有主链路的稳定性、边界和上线条件补齐”。
- 评分边界现在已经推进到配置化步进校验，P0 剩余重点进一步收敛到小程序真机联调与发布前自检。
- 运行时自检已经可以脚本化执行，后续上线准备可以更快从“人工判断”转向“脚本检查 + checklist”。
- 发布清单首版已经成型，后续可以围绕 checklist 继续做发布前真机冒烟和配置核对，而不是零散推进。

### 阶段 C：外部集成与上线收口

目标：

- 把当前骨架从“开发中产品”推进到“具备真实对外测试条件”

建议优先顺序：

1. 用真实 `js_code` 完成小程序换码联调
2. 发布前真机冒烟与配置核对
3. 保持运行时自检与发布清单可执行

完成标准：

- 小程序登录接口能用真实 `js_code` 换取正式会话
- 有明确上线 checklist，并能在发布前完成自检与真机冒烟

### 阶段 D：社交与差异化能力

目标：

- 开始做真正拉开产品差异的关系链和互动玩法

建议优先顺序：

1. 双人匹配链路
2. 匹配报告
3. 时光胶囊
4. AI 工作台与 QA 增强

完成标准：

- 至少 1 条匹配链路可完整跑通
- 高差异化玩法不再只是文档规划

## 8. 当前建议

这次阶段性复盘后的明确建议是：

1. P0 只做收尾，不再横向扩张范围。
2. 下一阶段先补“评分参数边界 / 小程序真机联调”。
3. 成长体系已经有首版，不必再把它当主轴扩张，除非它直接影响留存验证。
4. 匹配、时光胶囊、更多社交玩法先不要提前启动。
5. 当前最合理的推进顺序应调整为：
   1. 剩余题型与评分增强
   2. 小程序真实换码联调
   3. 发布前自检与真机冒烟
   4. P0 收口完成后，再进入 P1 / P2

换句话说，项目现在最需要的不是“再加功能”，而是把已经做出来的主链路收成一个更稳、更能上线、更便于内测验证的正式产品骨架。
