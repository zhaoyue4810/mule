# 心测项目状态板

这份文档用于持续记录项目当前做到哪里、已经完成什么、还缺什么，以及接下来最合理的推进顺序。

最后更新时间：2026-03-14（成长体系首版更新：勋章 / 日历 / 灵魂碎片）

## 1. 阶段性结论

当前项目已经从“原型资料 + 技术方案”推进到“可运行、可演示、可持续扩展的 MVP 骨架”阶段。

可以明确认为已经完成的，是以下 5 个里程碑：

1. 内容运行态底座完成：YAML、数据库模型、同步、导入、草稿、发布都已经建立。
2. 用户主链路打通：测试列表 -> 详情 -> 答题 -> 提交 -> 报告 -> 个人中心回看已经可运行。
3. 报告系统基础成型：结构化报告、AI 状态流、fallback 文案、Prompt 模板、Gateway 抽象已经建立。
4. 超级管理员运营底座形成：内容版本管理、发布控制、AI 任务查看、Prompt 模板编辑都已有 API 能力。
5. 身份体系底座形成：访客会话、手机号绑定、小程序登录预留、前端 token 会话已经建立。

当前项目还不能定义为“可上线正式版”，但已经明显越过 demo 阶段，进入“核心产品闭环已成型，接下来要做产品打磨与上线收口”的阶段。

当前整体判断：

- P0 可上线 MVP 完成度：约 `93%`
- 当前最接近的目标：补齐剩余低频题型、增强评分、继续把成长体系做成更完整的回访与留存层
- 当前最主要的缺口：评分细节、成长体系、外部供应商收口、上线运维
- 当前路线调整：下一阶段优先补“产品核心体验”，短信 / 微信 / 运维等外部集成放到后置收口

## 5. 当前主要风险与假设

### 风险

- 产品文档中的首批 8 套测试，与技术文档 / demo 当前基线不完全一致。
- H5 已切到正式 token 会话，但手机号正式短信、小程序真实换码和跨端账号归并仍未最终收口。
- 手机号验证码链路已经打通，但短信当前仍是 `mock` provider，正式投产前必须切真实供应商。
- 评分和报告目前能跑，但还没有覆盖更复杂题型、更多统计分层和异常输入边界。
- 当前已补提交完整性校验、题型级答案形态校验、数值范围校验，并支持 `rank / plot2d / colorpick` 复杂协议，但还缺更细的参数约束和防作弊策略。
- 百炼和 Ark 都已完成真实 key 冒烟，但 AI 成功率 / 时延 / 降级策略还需要继续观察。
- 成长体系当前已有勋章、日历和碎片首版，但还缺“持续回访理由”的下一层，比如每日一问和分享反馈。

### 当前假设

- 继续以 `index.html + 技术文档` 作为产品功能基线。
- 当前阶段继续优先后端和主链路闭环，不优先投入独立后台前端。
- AI 先按“可替换供应商的生产底座”推进，再做真实联调和细化运营能力。
- H5 先走“访客登录 -> token 会话 -> 后续升级绑定手机号/微信”的路线；小程序接微信换码登录。
- 短信供应商优先按阿里云 SMS 接入，当前 `mock` 仅用于本地开发和自动化测试。

## 8. 当前建议

这次阶段性复盘后的明确建议是：

1. 现在不要急着收所有第三方供应商。
2. 下一阶段优先把“题型 / 评分 / 报告 / 成长首版”补成更完整的产品体验。
3. 外部集成收口放在成长首版之后再集中做，会更符合当前节奏。
4. 当前最值得继续做的顺序应调整为：
   1. 剩余题型与评分增强
   2. 报告稳定性与 AI 审计监控
   3. 成长体系首版（勋章 -> 日历 -> 碎片）
   4. 短信供应商与微信登录收口

换句话说，项目现在最需要的不是继续补外围集成，而是把已经形成的主链路打磨成更像正式产品的体验，再进入外部系统收口与上线准备。
