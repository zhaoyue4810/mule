## 4. 开发原则

1. 先做内容与数据，再做页面表现。
2. 每次只推进一个明确子域，避免大而散。
3. 先做 P0 上线闭环，再做 P1 / P2 差异化能力。
4. demo 只作为体验和数据来源，不直接复用原生 DOM 实现。
5. 所有高频业务能力优先做成可测试、可配置、可扩展的结构。
6. 后台内容管理按“超级管理员直控草稿与发布”设计，暂不引入审批流。

## 11. 风险与跟踪项

### 风险 1：测试基线可能切换

如果你后面决定采用产品文档中的另一组 8 套测试，我们需要重做内容基线映射。

### 风险 2：小程序端复杂交互降级

15 种题型不一定都能 1:1 还原，后续要允许分层交付。

### 风险 3：导入质量不稳定

docx / html 导入需要“规则解析 + 草稿映射”两层机制，不能直接跳过内容结构化。

### 风险 4：AI 集成过早会拖慢主线

在 P0 闭环形成前，AI 只做预留接口和兜底设计，不提前展开复杂联调。
