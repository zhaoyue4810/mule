# 心测项目开工说明

## 1. 先看结论

这个项目不是“把 `index.html` 拆文件”，而是把原型迁移成一个配置驱动的正式产品平台。最合适的起步方式不是直接写页面，而是先把下面 3 件事固定下来：

1. 确定以技术文档为主线的工程结构
2. 把 `index.html` 里的结构化数据抽出来
3. 让后端先具备“加载配置、暴露接口、校验内容”的能力

所以这次起步我先落了后端骨架、YAML 配置、校验脚本和 demo 抽取脚本；这一轮继续把 8 套测试的第一版 YAML 基线补齐了。

## 2. 三份源材料分别告诉了我们什么

### 技术文档

`xince-technical-design-final_121644ff.md` 已经给出了比较明确的目标架构：

- 前端：`uni-app + Vue 3 + TypeScript + Pinia`
- 后端：`FastAPI + SQLAlchemy + Celery`
- 配置策略：题型、题库、提示词都走 YAML
- 产品节奏：先 P0 闭环，再做匹配、勋章、日历、记忆系统

最重要的是它明确了开发优先级：

1. 先建 YAML 配置文件
2. 再建数据结构与 YAML 同步
3. 再建接口
4. 再建页面
5. 最后接 AI

### 产品文档

`xince-design-doc.docx` 说明了这个产品的体验基调和交互复杂度：

- 核心风格是“温暖专业主义”
- 一共有 15 种题型交互
- 重点体验包括灵魂画像、匹配、勋章、日历、小测吉祥物
- 页面已经覆盖首页、发现、个人中心、匹配中心、报告、画像、日历、设置等核心视图

### HTML demo

`index.html` 是当前最真实的“初始化数据源”和“交互参照物”：

- 已有 8 套测试定义
- 已覆盖 15 种题型
- 已包含 badge / match / soul / calendar / daily question / memory 等核心常量
- 已体现状态流转和页面路由结构

但技术文档里也说得很清楚：它应该被当作高保真原型和数据来源，而不是直接复用 DOM 代码。

## 3. 当前阶段最合理的目标

按照技术文档第 22 节，现在最适合做的是“阶段 1：脚手架与基础设施”，然后自然过渡到“阶段 2：内容模型与导入系统”。

### 阶段 1 目标

- 建立 `server/` 后端骨架
- 建立 `app/config/` YAML 目录
- 跑通 YAML 加载和基础 API
- 给后续 `yaml_sync_service` 留好扩展点

### 阶段 2 紧接着要做的事

- 把 8 套测试 YAML 与数据库模型打通
- 把导入、校验、同步机制做起来
- 固化“内容是配置源，数据库是运行态”的原则

## 5. 当前发现的一个需求风险

产品文档中的“8 套测试内容库”描述，与 `index.html` / 技术文档中的 8 套测试并不完全一致。

产品文档列举的是：

- 大五人格
- MBTI 快测
- 依恋风格
- 情绪色彩
- 压力弹性
- 社交人格
- 内心小孩
- 未来自我

而 `index.html` 和技术文档当前实现导向的是：

- 大五人格
- MBTI
- 九型人格
- 恋爱人格图鉴
- 闺蜜默契度
- 职业天赋探测
- CP 默契挑战
- 职场人格图鉴

这说明正式开发前，需要先确定“上线首批 8 套测试”到底以哪一版为准。现阶段我先以 `index.html + 技术文档` 作为执行基线，因为它们彼此一致，且更贴近现有可迁移代码。

## 7. 当前后台策略

当前内容后台按“超级管理员直控”推进，不做多角色审核流。也就是说：

1. 导入任务负责生成草稿
2. 草稿内容进入后台版本库
3. 超级管理员直接决定发布哪个版本

现阶段的目标不是做复杂审批，而是先把“草稿可管理、线上可切换、发布可回溯”这条 MVP 主链路稳定下来。

## 8. 下一步建议

按收益排序，下一轮最推荐直接做这 4 个子任务：

1. 把导入草稿补成可查看、可编辑的后台内容详情
2. 把发布版本与用户端读取接口打通
3. 开始搭 `apps/xince-app` 页面骨架
4. 再做答题引擎与评分链路

这样推进，后面每一层都会建立在稳定的数据模型上，不会返工。
